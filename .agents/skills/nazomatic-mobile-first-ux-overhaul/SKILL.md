# Nazomatic Mobile-First UX Overhaul

## 固定コンテキスト

- ターゲットユーザー: 20代〜30代の謎解き好き、ゲーム操作に慣れている層
- デバイス方針: スマホ優先（PCも崩さない）
- デザインルール:
  - メイン: `bg-gradient-to-b from-gray-900 to-gray-800 text-white`
  - アクセント: `purple-400`
- 既存システムのUIトーンとコンポーネントを優先して再利用する
- 画面説明の原則: 長文説明を常時表示しない。説明追加はTooltipを優先する

## Tooltip導入ガイド（説明文の扱い）

- 参考実装: `src/components/graphpaper/graph-paper-component.tsx`
- 目的: 画面の情報密度と認知負荷を下げるため、説明文を必要時のみ提示する
- 基本方針:
  - 1行で済む補助文はインラインで残してよい
  - 2行以上の使い方説明、例示、ルール説明はTooltipへ移す
  - 新しい説明文を追加する場合はまずTooltip化を検討する
- 実装方針:
  - `graph-paper-component.tsx` と同様に「短い補足文 + ?アイコン」の構成を基本にする
  - ヘルプ本体は初期非表示にして、トリガー操作時のみ表示する
  - モバイルでは touch、PCでは hover/focus で読めるようにする
  - 複雑な内容はヘルプボックス内に箇条書きで整理する
  - 短い補足は `@/components/ui/tooltip`、内容が多い場合は `graph-paper-component.tsx` のようなカスタム表示を使う
- スタイル方針:
  - Tooltip/ヘルプ本体はダークトーン（`bg-gray-900/95`、`border-gray-700`、`text-gray-100`）
  - 強調色に `text-purple-400` を使う
  - モバイルで見切れない配置と幅にする（`left/right` 指定や `max-w` を適切に設定）
- アクセシビリティ:
  - キーボードフォーカスで開けること
  - トリガー要素にラベル（`aria-label`）を付けること
  - 小さすぎるタップ領域を避けること

## 受け付ける最小入力

- 必須（どちらか1つ）:
  - 対象ルート（例: `/blank25`）
  - 対象ファイル（例: `src/components/blank25/problem-list.tsx`）
- 任意:
  - 追加要望（例: CTAを強調したい）
  - スクリーンショット

入力が不足する場合は、会話コンテキストの開いているファイルや対象機能から合理的に補完して進める。

## 実行手順

1. 対象画面と関連ファイルを特定する（ルート指定時は `src/app` と関連 `src/components` を探索）。
2. 現状UIを診断する。
3. 次の観点で問題を抽出する。
   - 情報設計
   - 認知負荷
   - 操作導線
   - UIの一貫性
   - モバイル適応
   - 視覚階層
   - アクセシビリティ
   - エラー/空状態
4. 常時表示の説明文を棚卸しし、Tooltipへ移すべき要素を特定する。
5. 改善案を3〜5件に絞る（影響が大きい順）。
6. 小手先の調整よりUX効果を優先し、必要なら大胆にレイアウト再設計する。
7. 提案で止めず、必ずコードを修正する。
8. スマホ表示を最優先で整え、PC表示の破綻を防ぐ。
9. 推測と事実を分けて説明する。

## 出力形式

A. 重大課題トップ3（影響度/頻度/修正コスト）  
B. 実装方針（根本改善の方針）  
C. 実装結果（変更ファイルごとの要点）  
D. マイクロコピー改善（Before/After）  
E. QAチェックリスト（スマホ/PC、アクセシビリティ、空状態、エラー状態）

## 重要ルール

- レビューのみで終了しない。必ず実装まで行う。
- カラー/トーンは固定コンテキストを厳守する。
- 既存のデザインシステムから大きく乖離する新規スタイルの乱立を避ける。
- 説明情報を増やすときはTooltip-firstで設計する。
