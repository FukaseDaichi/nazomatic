# design-sync NOTES — NAZOMATIC

Repo-specific gotchas for future re-syncs. Read this first.

## What this DS is

- NAZOMATIC is a **Next.js app**, not a published component library. The synced
  "design system" is the **shadcn/ui** primitive set under `src/components/ui/`
  (13 component families). There is no `dist/`, no package `exports`.
- Styling is **stock Tailwind utility classes** (`cssVariables: false`, baseColor
  gray). No custom theme colors. Component CSS must be **generated by Tailwind**,
  not scraped from a build.

## Build mechanics (why the config looks the way it does)

- **Entry**: `.design-sync/ds-entry.ts` is a hand-maintained barrel re-exporting
  every `src/components/ui/*` primitive via the `@/` alias. Passed as `--entry`.
  Two jobs: (1) makes `PKG_DIR` walk up to the repo root (the converter sets
  `PKG_DIR = node_modules/<pkg>` otherwise, which doesn't exist for an app repo);
  (2) puts every export — including compound sub-parts (CardHeader, DialogContent…)
  — into `window.NAZOMATIC` so previews can compose them. **Add a line here when a
  new ui primitive is synced.**
- **Cards = the 13 `componentSrcMap` heads only.** Sub-parts are in the bundle but
  don't get their own cards (componentSrcMap pins exactly the 13 family heads).
- **CSS**: `cfg.buildCmd` runs Tailwind over the repo (config content = whole
  `src/`) into `.design-sync/generated/ds.css`; `cfg.cssEntry` points there. The
  whole-`src` content scan guarantees the NAZOMATIC look classes used in preview
  wrappers (gradient bg, `purple-400`) are present. Re-run `buildCmd` before the
  converter on every re-sync.
- **Dark-first**: components default to light (`bg-white`); the NAZOMATIC look
  (`.dark` + `from-gray-900 to-gray-800` gradient + `purple-400` accent) lives in
  page/usage code, NOT in the primitives. Authored previews wrap content in a
  `.dark` + gradient container so the cards show the real intended appearance.

## Build/install

- node_modules is the app's own; all primitive deps (react, radix, cva, clsx,
  tailwind-merge, lucide-react, tailwindcss) present. `npm ci` skipped on first
  sync (deps already complete) — run it on a fresh clone.
- Run: `node .ds-sync/package-build.mjs --config .design-sync/config.json
  --node-modules ./node_modules --entry .design-sync/ds-entry.ts --out ./ds-bundle`

## Known render warns (re-syncs: check new warns against this list)

- `[FONT_MISSING] "Cambria"` — Cambria appears only in Tailwind's **default
  `font-serif` fallback stack** (`ui-serif, Georgia, Cambria, "Times New Roman"…`).
  It is NOT a NAZOMATIC brand font; the app ships no custom fonts. Accepted: designs
  render with system serif fallbacks. Do not chase — do not add `extraFonts`.
- `RENDER_BLANK` on a component currently shipping the **floor card** is expected
  (the typographic floor PNG is <5KB). Only a real concern once a preview is authored.

## CRITICAL styling lesson (drives the conventions header)

- **Brand color on Button/Badge backgrounds needs a `dark:` variant.** shadcn's
  default `Button`/`Badge` variants set `dark:bg-gray-50` (a light bg in dark mode).
  Since NAZOMATIC is dark-first, a bare `className="bg-purple-600"` is overridden by
  `dark:bg-gray-50` → the control renders WHITE. Always pair it:
  `bg-purple-600 hover:bg-purple-500 dark:bg-purple-600 dark:hover:bg-purple-500 text-white`.
  Text-only purple (`text-purple-400`) has no dark override and works as-is.
- **Overlay components (Dialog/Sheet/Select/Tooltip) portal to `document.body`**,
  outside any `.dark` wrapper, so previews call a `useDark()` effect that adds
  `.dark` to `document.documentElement`. Also: shadcn's portaled content sets a
  dark *background* (`dark:bg-gray-950`) but NOT a text color — in the real app the
  body provides `text-white`; in a preview you must add `text-gray-50`/`text-white`
  to the content or the text is invisible (dark-on-dark).
- Overlay `cardMode: single` + a viewport wider than the content's `max-w-lg`
  (512px) — use ≥600px width or the dialog is clipped/cramped.

## .d.ts contracts

- Synth-entry mode has no shipped `.d.ts`, so the ts-morph extractor produced
  `[key: string]: unknown` bags. Real, design-facing contracts are hand-written in
  `cfg.dtsPropsFor` for all 13 heads (variant/size enums, Radix Root props). **Keep
  these in sync with the source** when a component's API changes upstream.

## Re-sync risks

- `ds-entry.ts` and `componentSrcMap` are hand-maintained — a new ui primitive
  won't sync until both are updated.
- The generated CSS depends on the app's `tailwind.config.ts` content globs; if
  those change, the synced CSS coverage changes with them.
