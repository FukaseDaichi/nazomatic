# BLANK25 チーム戦ルール説明サイト 実装方針（2026-03-09）

## 1. 目的

- `/blank25` 配下に、既存の BLANK25 を使ったチーム戦の遊び方を説明する特設ページを追加する。
- ただルールを並べるのではなく、「どこまで隠すか」を攻めるチキンレースとして面白さが伝わる構成にする。
- 既存のゲーム画面、作問モード、パーティ得点ボードへの導線を 1 か所にまとめる。

## 2. スコープ

### 2.1 対象

- 新規ルール説明ページの追加
- `/blank25` または `/blank25/party` からの導線追加
- チーム戦ルールの整理
- モバイル優先のインタラクティブな説明ギミック
- 既存 `/blank25` と `/blank25/party` への CTA 設計

### 2.2 対象外

- チーム戦そのものの自動進行
- 別端末同期
- チーム管理やターン管理のサーバー保存
- 制限時間の厳密な自動計測
- 勝敗の自動確定

このページは「遊び方を理解し、現場で回せるようにする」ための説明サイトであり、ゲーム進行システムそのものは持たせない。

## 3. チーム戦ルール整理

### 3.1 勝負の核

- 得点を競うゲームである。
- 面白さの中心は「チームメイトがギリギリ解答できそうなラインまで隠す」チキンレースにある。
- 正解できれば、残りパネル数がそのまま得点になる。
- 欲張りすぎて不正解なら、そのラウンドは 0 点になる。

### 3.2 基本フロー

1. 参加者を集め、チームに分ける。
2. 各チームで、そのラウンドの出題者を 1 人決める。
3. 出題者は作問モードを開き、制限時間内にパネルを隠して問題を作る。
4. 回答者は制限時間内に答える。
5. 正解なら「残りパネル数」を得点として加算する。
6. 不正解なら 0 点とする。

### 3.3 全滅時ルール

- そのラウンドで全チームが解けなかった場合はやり直しにする。
- やり直し時は作問し直す。
- ただし、前回までに開いたパネルは必ず開いたままにする。

このルールによって、全滅のたびに盤面が少しずつ甘くなり、無茶な作問を続けるほど高得点が取りにくくなる。

## 4. 現行仕様との整合

- 現行の `/blank25/[problemId]?mode=sakumon` は、正解時スコアを `表示数` で計算している。
- チーム戦で言う `残りパネル数` は、この `表示数` と同義として扱う。
- そのため、ルール説明サイトの得点説明は `残りパネル数` を基本表現にしつつ、必要に応じて `表示数` を補足として併記する。

説明サイト公開時は、「残りパネル数」と「表示数」が別物に見えないようにラベルを統一する。

## 5. 追加ルート方針

- 新規ルートは `/blank25/party/rules` を推奨する。
- 理由は、通常の問題プレイ `/blank25/[problemId]` と責務を分けつつ、既存のパーティ得点ボード `/blank25/party` と文脈を合わせやすいため。
- ルール説明ページは「遊び方を理解する場所」、`/blank25/party` は「得点を管理する場所」として役割を分ける。

想定ファイル:

- `src/app/(blank25)/blank25/party/rules/page.tsx`
- `src/components/blank25/team-battle-rules.tsx`
- `src/components/blank25/team-battle-chicken-meter.tsx`
- `src/components/blank25/team-battle-round-flow.tsx`
- `src/components/blank25/team-battle-retry-simulator.tsx`
- `src/components/blank25/problem-list.tsx` または `src/components/blank25/party-scoreboard.tsx` の導線追加

## 6. UX コンセプト

- ベースは既存ルールどおり `bg-gradient-to-b from-gray-900 to-gray-800 text-white` を維持する。
- アクセントは `purple-400` を主役にする。
- エンタメ性は新配色ではなく、コピー、演出、段階的な見せ方、盤面モチーフで出す。
- 1 セクション 1 メッセージを徹底し、モバイルでも読み切れる構成にする。
- 長い説明文ではなく、要点は大きなカードと短い見出しで見せる。
- 補足は Tooltip や小さな注釈に寄せ、本文を重くしすぎない。

## 7. 情報設計

### 7.1 ページ構成案

1. ヒーロー
2. このゲームの面白さ
3. 1 ラウンドの流れ
4. 得点の決まり方
5. 全滅したらどうなるか
6. 実際に使うページへの導線
7. 運用メモ

### 7.2 各セクションの役割

| セクション | 役割 | 主要 UI |
| --- | --- | --- |
| ヒーロー | チーム戦の一言理解 | 大見出し、短いコピー、CTA |
| このゲームの面白さ | チキンレース性の理解 | インタラクティブな攻め度メーター |
| 1 ラウンドの流れ | 手順の誤解防止 | タイムライン、役割カード |
| 得点の決まり方 | 正解 / 不正解の差を即理解 | ミニ盤面、得点ラベル |
| 全滅したらどうなるか | 複雑ルールの可視化 | リトライ用の盤面シミュレーター |
| 実際に使うページへの導線 | 行動開始 | `/blank25` と `/blank25/party` の CTA |
| 運用メモ | 現場の混乱防止 | 箇条書き、Tooltip |

## 8. エンタメギミック方針

### 8.1 中核ギミック: チキンメーター

- 25 マスのミニ盤面を使い、隠す枚数と残りパネル数の関係を同時に見せる。
- 隠すほど攻め度は上がるが、得点として残るパネル数は減っていくことを一目で伝える。
- ただし、コピーは同時に「安全」「ギリギリ」「欲張りすぎ」のように変化させる。
- 数字だけでなく、心理的な危なさを見せることで、この遊びの面白さを一発で伝える。

表示例:

- 5 枚隠し: 残り 20 点、まだ安全圏
- 11 枚隠し: 残り 14 点、だいぶチキンレース
- 16 枚隠し: 残り 9 点、その 1 枚、本当に味方は耐えられる？

### 8.2 演出ギミック: 実況風ラウンド進行

- `出題者、作問スタート`
- `味方がざわつくラインまで隠せ`
- `正解なら高得点、外したら 0`

このような実況風の短いコピーを、ラウンド手順のカードに乗せる。

説明文を増やすより、場の空気が伝わる短い文でテンポよく読ませる方が相性が良い。

### 8.3 説明ギミック: 全滅リトライシミュレーター

- 1 回目の挑戦で全チーム失敗した状態をミニ盤面で再現する。
- `開いたまま残るパネル` を視覚的に固定表示する。
- 2 回目は「もう同じ強気は通せない」ことを見せる。

このギミックは、文章だけだと伝わりづらい「前回のパネルは必ず空いたまま」の理解に最も効く。

## 9. コピー方針

- 断定的で短いコピーを使う。
- 説明調より、司会や実況に近いテンポを優先する。
- ただし、ルール解釈に影響する箇所は比喩でぼかさず明記する。

見出し例:

- `その 1 枚、隠す？`
- `味方が解けるギリギリを攻めろ`
- `外したら 0 点`
- `全滅したら、盤面はもう甘くならない`

## 10. 実装構成案

### 10.1 ページ責務

- `page.tsx` は `ArticleHeaderComponent` とメタ情報を持つ薄いサーバーコンポーネントにする。
- 本体はクライアントコンポーネント `team-battle-rules.tsx` に寄せる。
- 盤面を使うギミックだけを小さな子コンポーネントへ分離する。

### 10.2 コンポーネント責務

- `team-battle-rules.tsx`
  - 全体レイアウト
  - 各セクションの並び
  - CTA まとめ
- `team-battle-chicken-meter.tsx`
  - 隠し枚数に応じた盤面表示
  - リスク文言の切り替え
- `team-battle-round-flow.tsx`
  - チーム分けから採点までの流れ
  - 出題者 / 回答者の役割表示
- `team-battle-retry-simulator.tsx`
  - 全滅時の再作問ルールの可視化

### 10.3 実装上の注意

- 盤面デモは説明用であり、実ゲームの状態保存や問題取得 API には依存しない。
- 説明ページのギミックはローカル state のみで完結させる。
- モバイルでは縦積み優先にし、盤面は正方形を維持する。
- クリック対象はすべて `button` を使い、タップしやすいサイズを確保する。
- 説明ページに入力フォームを増やさない。もし入力が必要になった場合は `text-base` 以上を明示する。

## 11. 導線方針

- `/blank25` の一覧ページに「チーム戦ルール」カードを追加する。
- `/blank25/party` からも「遊び方を見る」で遷移できるようにする。
- ルール説明ページの最後に、次の 2 つの CTA を並べる。

1. `問題を選んで作問する` -> `/blank25`
2. `得点ボードをひらく` -> `/blank25/party`

これにより、理解してからすぐ遊び始められる流れを作る。

## 12. 段階的な実装手順

1. ルール説明ページのルートと大枠レイアウトを追加する。
2. ヒーロー、ラウンドフロー、CTA の静的セクションを先に実装する。
3. チキンメーターを追加する。
4. 全滅リトライシミュレーターを追加する。
5. `/blank25` と `/blank25/party` に導線を追加する。
6. 得点ラベルと説明文を `残りパネル数` / `表示数` で矛盾なく統一する。

## 13. 受け入れ条件

1. ルール説明ページだけ見れば、チーム分け、出題者、回答者、得点、全滅時ルールまで理解できる。
2. 「残りパネル数が得点」「不正解は 0 点」が視覚的に伝わる。
3. 「全チーム失敗時は前回のパネルを開いたまま再作問」が、文章だけでなく UI でも理解できる。
4. モバイルでも縦スクロールで読み切れ、盤面デモが破綻しない。
5. ページ末尾から `/blank25` と `/blank25/party` にすぐ遷移できる。
6. 説明内容と実ゲームの得点定義に矛盾が残らない。
